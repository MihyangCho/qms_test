## 1. 基本動作

### 1-1. LINE認証ログイン（REQ-F-001）

- LINEアプリ内ブラウザからLIFF（LINE Front-end Framework）経由でアクセスする
- 初回アクセス時、LINEログイン認証画面が表示される
- 認証成功後、LINEのユーザーID・表示名・メールアドレス（許可時）を取得する
- 取得したユーザーIDをもとにシステム内のユーザーレコードを照合し、未登録であれば新規作成する
- 2回目以降のアクセスでは、有効なアクセストークンが存在する場合は認証画面をスキップする

### 1-2. 予約登録（REQ-F-002）

- 予約入力画面 → 予約確認画面 → 予約完了画面の3ステップで遷移する
- 予約確認画面で「予約する」ボタンを押下すると、サーバー側で以下の処理が実行される：
  1. 対象時間帯の定員枠を確認し、空きがあることを検証する
  2. 予約レコードをDBに挿入する（ステータス：`予約済`）
  3. 予約番号を自動採番する
  4. チェックイン用QRコードを生成する
  5. LINE Messaging APIで予約完了通知を送信する
- 上記処理はトランザクション内で実行し、途中でエラーが発生した場合はロールバックする

### 1-3. 予約変更（REQ-F-008）

- 予約詳細画面から「変更する」ボタンで予約変更画面に遷移する
- 変更可能な項目：来場日、来場時間、同伴者人数
- 変更不可の項目：氏名、生年月日、メールアドレス
- 変更確定時、変更先の時間帯の定員枠を確認し、空きがある場合のみ変更を受け付ける
- 変更成功後、QRコードは再発行せず既存のものを維持する
- LINE Messaging APIで変更完了通知を送信する

### 1-4. 予約完了通知（REQ-F-005）/ 変更完了通知（REQ-F-010）

- LINE Messaging APIのプッシュメッセージで送信する
- 通知メッセージに含める内容：
  - 予約番号
  - 来場日・来場時間
  - 同伴者人数
  - QRコード画像（予約完了通知のみ）
- 送信失敗時はリトライを1回行い、それでも失敗した場合はエラーログに記録する（予約自体は成立させる）

### 1-5. QRコード発行（REQ-F-006）

- 予約番号をもとにQRコードを生成する
- QRコードのペイロードは予約番号を暗号化した文字列とする
- 画像形式：PNG、サイズ：300x300px
- 生成したQRコード画像はサーバー上に保存し、予約詳細画面およびLINE通知から参照する

### 1-6. 予約番号自動採番（REQ-F-007）

- 採番ルール：`EVT-YYYYMMDD-NNNN`
  - `YYYY`：予約作成年
  - `MM`：予約作成月
  - `DD`：予約作成日
  - `NNNN`：当日内の連番（0001〜9999）
- 同時リクエスト時の重複を防ぐため、DBシーケンスまたはアトミックなカウンターで採番する

## 2. 時間制御

### 2-1. 予約受付期間

| 制御項目 | ルール |
|----------|--------|
| 予約開始 | イベント開催日の30日前 00:00:00 |
| 予約締切 | イベント開催日の前日 23:59:59 |
| タイムゾーン | JST（Asia/Tokyo） |

- 受付期間外にアクセスした場合、予約入力画面に「現在予約受付期間外です」と表示する
- 予約入力中に締切時刻を超過した場合、確定ボタン押下時にサーバー側で再チェックし、期間外であればエラーを返す

### 2-2. 変更・キャンセル期限（REQ-F-009）

| 制御項目 | ルール |
|----------|--------|
| 変更可能期限 | 来場日の前日 23:59:59 |
| キャンセル可能期限 | 来場日の前日 23:59:59 |
| 判定基準 | サーバー側の現在時刻（JST）で判定 |

- 期限内：変更/キャンセルボタンを表示する
- 期限超過：変更/キャンセルボタンを非表示にし、「変更・キャンセル期限を過ぎています」と表示する
- 期限判定はフロント側の表示制御に加え、API側でも必ず再判定する（クライアント時刻の改ざん対策）

### 2-3. リマインダー通知

| 制御項目 | ルール |
|----------|--------|
| 送信タイミング | 来場日の前日 18:00:00（JST） |
| 対象 | ステータスが`予約済`の予約 |
| 実行方式 | バッチ処理（cron等） |

- キャンセル済の予約にはリマインダーを送信しない

## 3. 入力欄のバリデーションルール

### 3-1. 予約入力画面の各項目（REQ-F-003）

| 項目 | 必須 | 形式 | バリデーションルール |
|------|------|------|---------------------|
| 氏名 | ○ | テキスト | 1〜50文字。前後の空白はトリムする |
| 生年月日 | ○ | 日付 | 過去の日付であること。未来日は不可 |
| 来場日 | ○ | 日付選択 | 予約受付期間内の日付であること |
| 来場時間 | ○ | 時間帯選択 | 選択した来場日で空きのある時間帯のみ選択可能 |
| メールアドレス | ○ | テキスト | RFC 5322準拠の形式チェック。255文字以内 |
| 電話番号 | △ | テキスト | 10〜11桁の数字（ハイフンは自動除去）。先頭が0であること |
| 同伴者人数 | △ | 数値選択 | 0〜4の整数。未選択時は0として扱う。上限値は管理画面から変更可能とする |

### 3-2. バリデーション実行タイミング

- **フロント側**：各項目のフォーカスアウト時にリアルタイムバリデーションを行う
- **サーバー側**：フォーム送信時に全項目を再バリデーションする（フロント側のバリデーションは信頼しない）
- フロントとサーバーで同一のバリデーションルールを適用する

### 3-3. 来場時間の選択肢制御

- 来場日を選択した時点で、対象日の時間帯一覧をAPIから取得する
- 定員に達した時間帯は「満員」と表示し、選択不可にする
- 時間帯の選択肢は管理画面で設定された値を使用する

## 4. エラーハンドリング

### 4-1. バリデーションエラー

| エラー種別 | 表示方法 | メッセージ例 |
|------------|----------|-------------|
| 必須項目未入力 | 対象項目の直下に赤字表示 | 「氏名を入力してください」 |
| 形式不正 | 対象項目の直下に赤字表示 | 「メールアドレスの形式が正しくありません」 |
| 範囲外 | 対象項目の直下に赤字表示 | 「同伴者は0〜4名で入力してください」 |

- 複数のエラーがある場合、全エラーを同時に表示する（1つずつではない）
- エラー発生時、画面は最初のエラー項目までスクロールする

### 4-2. 業務エラー

| エラー条件 | HTTPステータス | 画面表示 |
|------------|---------------|----------|
| 定員超過（予約確定時） | 409 Conflict | 「選択された時間帯は満員になりました。別の時間帯をお選びください」 |
| 変更・キャンセル期限超過 | 403 Forbidden | 「変更・キャンセル期限を過ぎています」 |
| 予約受付期間外 | 403 Forbidden | 「現在予約受付期間外です」 |
| 同一日に予約済み | 409 Conflict | 「同じ日に既に予約があります」 |
| 予約が存在しない | 404 Not Found | 「予約が見つかりません」 |

### 4-3. システムエラー

| エラー条件 | HTTPステータス | 画面表示 |
|------------|---------------|----------|
| サーバー内部エラー | 500 Internal Server Error | 「システムエラーが発生しました。しばらく時間をおいて再度お試しください」 |
| LINE API通信エラー | （内部処理） | 予約処理は成立させ、通知送信のみリトライ。ユーザーには通知遅延の旨を画面表示 |
| DB接続エラー | 503 Service Unavailable | 「現在サービスが一時的にご利用いただけません」 |
| タイムアウト（30秒） | 504 Gateway Timeout | 「処理がタイムアウトしました。再度お試しください」 |

### 4-4. エラーログ

- 全エラーをサーバーログに記録する
- ログ項目：タイムスタンプ、リクエストID、ユーザーID、エラー種別、エラー詳細、リクエストパラメータ（個人情報はマスキング）
- 500系エラーはアラート通知の対象とする

## 5. 条件入力時のDB上データ変化

### 5-1. テーブル構成（関連テーブル）

| テーブル名 | 概要 |
|-----------|------|
| users | LINEユーザー情報 |
| reservations | 予約情報 |
| time_slots | 時間帯枠・定員管理 |
| notification_logs | 通知送信履歴 |

### 5-2. 予約登録時

**操作：** 予約入力画面で全項目を入力し「予約する」を押下

| テーブル | 操作 | 変化内容 |
|---------|------|---------|
| users | INSERT or UPDATE | 初回：新規レコード作成（line_user_id, display_name, email）。2回目以降：last_login_atを更新 |
| reservations | INSERT | status=`予約済`, reservation_no=自動採番, qr_code_url=生成パス, created_at=現在時刻 |
| time_slots | UPDATE | remaining_capacity = remaining_capacity - (1 + 同伴者人数) |
| notification_logs | INSERT | type=`予約完了`, status=`送信済` or `送信失敗`, sent_at=現在時刻 |

### 5-3. 予約変更時

**操作：** 予約詳細画面で来場日・来場時間・同伴者人数を変更し「変更する」を押下

| テーブル | 操作 | 変化内容 |
|---------|------|---------|
| reservations | UPDATE | visit_date, visit_time, companion_count を更新。updated_at=現在時刻 |
| time_slots（変更前） | UPDATE | remaining_capacity = remaining_capacity + (1 + 変更前の同伴者人数)　※枠を戻す |
| time_slots（変更後） | UPDATE | remaining_capacity = remaining_capacity - (1 + 変更後の同伴者人数)　※枠を消費 |
| notification_logs | INSERT | type=`変更完了`, status=`送信済` or `送信失敗` |

### 5-4. 予約キャンセル時

**操作：** 予約詳細画面で「キャンセルする」を押下し、確認ダイアログで「はい」を選択

| テーブル | 操作 | 変化内容 |
|---------|------|---------|
| reservations | UPDATE | status=`キャンセル済`, cancelled_at=現在時刻。レコードは物理削除しない |
| time_slots | UPDATE | remaining_capacity = remaining_capacity + (1 + 同伴者人数)　※枠を戻す |
| notification_logs | INSERT | type=`キャンセル完了`, status=`送信済` or `送信失敗` |

### 5-5. チェックイン時

**操作：** 管理者がQRコードを読み取り

| テーブル | 操作 | 変化内容 |
|---------|------|---------|
| reservations | UPDATE | status=`来場済`, checked_in_at=現在時刻 |

### 5-6. ステータス遷移図

```
予約済 ──→ 来場済（チェックイン）
  │
  └──→ キャンセル済（キャンセル操作）
```

- `来場済` → 他ステータスへの遷移は不可
- `キャンセル済` → 他ステータスへの遷移は不可（再予約は新規予約として扱う）
